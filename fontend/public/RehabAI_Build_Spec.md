# RehabAI — Full Build Specification (for Claude Code)

> เอกสารนี้เขียนเพื่อให้ Claude Code (หรือ dev ทั่วไป) ใช้เป็น source of truth ในการลงมือพัฒนาเว็บแอปจริง อ้างอิงจาก UI mockup ที่ออกแบบไว้ก่อนหน้า ครอบคลุมทุกหน้า ทุกฟิลด์ ทุก flow และโครงสร้างข้อมูลที่แนะนำ

---

## 0. Tech Assumptions

- **Frontend:** React (หรือเทียบเท่า) + component-based architecture, responsive desktop-first (ดูจาก mockup เป็นเว็บ ไม่ใช่มือถือ)
- **Backend/DB:** Firebase — Authentication (email/password) + Firestore (ดูโครงสร้าง collection ด้านล่าง) + Cloud Storage (รูปโปรไฟล์)
- **Realtime pose/voice analysis:** ต้องใช้ WebRTC (เข้าถึงกล้อง/ไมค์ของ browser) + pose estimation model (เช่น MediaPipe Pose หรือเทียบเท่า) รันฝั่ง client
- **SMS:** ผูกกับผู้ให้บริการ SMS gateway (เช่น Twilio, Thai SMS provider) ผ่าน Cloud Function — **ไม่ใช้อีเมลสำหรับการแจ้งเตือน**
- **โดเมน:** แอปผู้ป่วยอยู่ที่ `rehabai.app`, Therapist Portal อยู่คนละโดเมนย่อย `therapist.rehabai.app` (คนละ auth context, คนละ role)
- **กลุ่มผู้ใช้เป้าหมาย:** ผู้ป่วยพาร์กินสันระยะ 1-2 (Hoehn & Yahr) เท่านั้น ที่ยังช่วยเหลือตนเองได้ — **ไม่มีฟีเจอร์ fall-risk screening หรือ caregiver mode** ในขอบเขตนี้

## 1. Design System

### 1.1 Color tokens (CSS variables)
```css
--bg: #FAF8F4;            /* พื้นหลังหลักของแอป */
--surface: #FFFFFF;       /* พื้นผิวการ์ด/พาเนล */
--teal-900: #1F4A40;      /* ข้อความหัวเรื่อง, sidebar, ปุ่มหลัก (dark) */
--teal-800: #28584C;
--teal-700: #2F6F62;      /* สีปุ่มหลัก (primary action) — กลุ่ม LSVT BIG */
--teal-500: #4E9484;      /* กราฟ/แถบสีอ่อนกว่า */
--teal-100: #DCEEE8;      /* พื้นหลังไอคอน, badge, สถานะ stretch */
--coral-700: #B9542A;     /* สีกลุ่ม LSVT LOUD และ badge "balance" */
--coral-100: #FBE7D9;
--text-primary: #22302C;
--text-secondary: #5B6B66;
--text-muted: #8E9B96;
--border: #E6E0D4;
--cam-bg: #152622;        /* พื้นหลังพาเนลกล้อง/ไมค์ระหว่างฝึกจริง */
--danger: #D9483E;        /* ปุ่ม/แถบอันตราย (SOS, ลบบัญชี) */
```

**หลักการใช้สี:** teal = LSVT BIG / การกระทำหลัก, coral = LSVT LOUD / balance, เหลือง (#FDF3D9 + #9A6B0A) = คำเตือน/รอดำเนินการ, แดง = อันตราย/ฉุกเฉิน, เขียวอ่อน (#E6F0E1 + #3B6D11) = สำเร็จ/อนุมัติแล้ว

### 1.2 Typography
- Heading font: **Poppins** (weight 500, 600)
- Body font: **Inter** (weight 400, 500, 600) — fallback **Sarabun** สำหรับข้อความไทยล้วน
- Monospace (ตัวเลขจับเวลา, รหัส): **JetBrains Mono** (weight 500)
- ขนาดตัวอักษรสำหรับผู้สูงวัย: body 13-15px เป็นอย่างต่ำ, heading 18-28px, ปุ่มไม่ต่ำกว่า 13px และ padding อย่างน้อย 11-14px แนวตั้ง

### 1.3 Layout pattern
- **Auth pages** (Login/Register): split-screen — แบรนด์แพนเนลสีเขียวเข้ม (`teal-900`) ฝั่งซ้าย 38% + ฟอร์มฝั่งขวา
- **App pages**: sidebar ซ้าย 220px (พื้นหลัง `teal-900`) + main content (พื้นหลัง `bg`) — topbar สูง ~60px ด้านบนของ main content
- Border radius มาตรฐาน: การ์ด 12-14px, ปุ่ม 8-10px, avatar/badge วงกลมเต็ม

## 3. Page-by-Page Specification

### 3.1 `/login` (ผู้ป่วย)
**Layout:** split-screen
**ฝั่งซ้าย (brand panel):** โลโก้ RehabAI, หัวข้อ "ฟื้นฟูการเคลื่อนไหวและการพูด ในที่เดียว", คำอธิบายสั้น, footer เครดิตโครงการ
**ฝั่งขวา (form):**
- ลิงก์เล็กมุมขวาบน: `เป็นนักกายภาพบำบัด?` → ลิงก์ไป `therapist.rehabai.app/login`
- หัวข้อ "ยินดีต้อนรับกลับ"
- Field: อีเมล (type=email, icon=envelope)
- Field: รหัสผ่าน (type=password, icon=lock)
- Row: checkbox "จดจำฉันไว้" (ซ้าย) + ลิงก์ "ลืมรหัสผ่าน?" (ขวา)
- ปุ่มหลัก: "เข้าสู่ระบบ" (submit → Firebase Auth signIn)
- Divider "หรือ"
- ข้อความ: "ยังไม่มีบัญชี? สมัครสมาชิก" → ลิงก์ `/register`

**Validation:** อีเมลต้องเป็นรูปแบบถูกต้อง, รหัสผ่านห้ามว่าง, แสดง error message ใต้ field ที่ผิด (ไม่ใช้ alert popup — ผู้สูงวัยอ่านยาก)

---

### 3.2 `/register` (ผู้ป่วย)
**หลักการสำคัญ: ฟอร์มต้องสั้นที่สุด** (ลดภาระผู้สูงวัย/อาการมือสั่น) — ฟิลด์เบอร์ฉุกเฉินถูกย้ายออกไปเป็นขั้นตอนแยกหลังสมัครสำเร็จแล้ว (ดู 3.3)

**ฟิลด์ (เรียงตามลำดับ):**
1. ลิงก์เล็ก "เป็นนักกายภาพบำบัด?" มุมขวาบน → `therapist.rehabai.app/register`
2. ชื่อ-นามสกุล + อีเมล (2 คอลัมน์)
3. รหัสผ่าน + ยืนยันรหัสผ่าน (2 คอลัมน์)
4. ระยะอาการของโรคพาร์กินสัน (Hoehn & Yahr) — **select จำกัดแค่ 2 ตัวเลือก**: "ระยะ 1 — มีอาการข้างเดียว ยังช่วยเหลือตนเองได้เต็มที่", "ระยะ 2 — มีอาการสองข้าง ยังทรงตัวและช่วยเหลือตนเองได้"
   - Helper text ใต้ field: "RehabAI ออกแบบมาสำหรับผู้ป่วยระยะ 1-2 ที่ยังช่วยเหลือตนเองได้ หากมีอาการรุนแรงกว่านี้ แนะนำให้ปรึกษานักกายภาพบำบัดก่อนใช้งาน"
5. Checkbox: "ฉันยอมรับข้อตกลงการใช้งานและนโยบายความเป็นส่วนตัว" (required)
6. ปุ่ม "สมัครสมาชิก" → สร้างบัญชี Firebase Auth + เอกสาร `users/{uid}` → redirect ไป `/onboarding/emergency-contact`
7. ลิงก์ "มีบัญชีอยู่แล้ว? เข้าสู่ระบบ"

**Validation:** รหัสผ่านสองช่องต้องตรงกัน, ต้องติ๊กยอมรับเงื่อนไขก่อน submit ได้

### 3.3 `/onboarding/select-therapist` — เลือกนักกายภาพบำบัดของคุณ (tap-only, ไม่มีการพิมพ์)
แสดงทันทีหลังสมัครสมาชิกสำเร็จ **ก่อน** ขั้นตอนเพิ่มเบอร์ฉุกเฉิน (3.3b)

**หลักการสำคัญ:** สมมติฐานของระบบคือ **ผู้ป่วยทุกคนมาใช้แอปนี้เพราะนักกายภาพบำบัดแนะนำ** หมายความว่าผู้ป่วยเคยพบนักกายภาพบำบัดตัวจริงที่โรงพยาบาลมาแล้วอย่างน้อย 1 ครั้ง — ความสัมพันธ์นี้เองคือสิ่งที่ยืนยันความน่าเชื่อถือ ไม่ต้องมีขั้นตอนขอรหัสเชื่อมต่อหรือรออนุมัติแบบเดิมอีกต่อไป

**ฟอร์มเป็น cascading dropdown 3 ขั้น กดเลือกทั้งหมด ห้ามให้พิมพ์เอง** (ป้องกันการกรอกผิดพลาดในผู้สูงวัย) — ทั้ง 3 ขั้นใช้ **dropdown (select) สไตล์เดียวกัน** (pill กลม มีไอคอนซ้าย + chevron ขวา เหมือนฟิลด์ "สังกัดโรงพยาบาล" ในหน้าสมัครนักกายภาพบำบัด 4.2) **หัวข้อแต่ละขั้นเป็นข้อความเรียบง่าย ("เลือกภาค" / "เลือกโรงพยาบาล" / "เลือกนักกายภาพบำบัด") ไม่ต้องเขียนกำกับว่า "(เฉพาะ...)" ในหัวข้อ** — การกรองต้องเกิดจากพฤติกรรมจริงของ dropdown (options เปลี่ยนตามค่าที่เลือกไว้ก่อนหน้า) ไม่ใช่แค่ข้อความอธิบาย:

1. **เลือกภาค** — dropdown 1 ค่า: ภาคกลาง, ภาคเหนือ, ภาคตะวันออกเฉียงเหนือ, ภาคใต้, ภาคตะวันออก, ภาคตะวันตก
2. **เลือกโรงพยาบาล** — dropdown ที่ **options เปลี่ยนแบบเรียลไทม์ทันทีที่เปลี่ยนภาคในขั้นที่ 1** โดยแสดงเฉพาะโรงพยาบาลในภาคนั้น (filter จาก `hospitals` collection ด้วย field `region`) ควรมีฐานข้อมูลโรงพยาบาลครอบคลุมทุกภาคอย่างน้อยภาคละ 5-13 แห่ง (ไม่ใช่แค่ภาคกลาง) รวมทั้งโรงพยาบาลรัฐและเอกชนที่ร่วมโครงการ
3. **เลือกนักกายภาพบำบัด** — dropdown ที่ **options เปลี่ยนแบบเรียลไทม์ทันทีที่เปลี่ยนโรงพยาบาลในขั้นที่ 2** แสดงเฉพาะนักกายภาพบำบัดที่สังกัดโรงพยาบาลนั้น **และมีสถานะ `verificationStatus: "active"` เท่านั้น** (นักกายภาพบำบัดที่ยังไม่ผ่านการยืนยันจะไม่ปรากฏใน option) — แต่ละ option แสดง "ชื่อ — ตำแหน่ง (แผนก)" แล้วเลือก 1 คน

**ทันทีที่เลือกนักกายภาพบำบัด:**
- แสดงข้อความยืนยันสีเขียว: "เชื่อมต่อกับ {ชื่อนักกายภาพบำบัด} เรียบร้อยแล้ว เนื่องจากคุณเลือกนักกายภาพบำบัดที่มีตัวตนยืนยันแล้วในระบบ ระบบจึงเชื่อมต่อบัญชีให้ทันที ไม่ต้องรออนุมัติ"
- สร้าง doc ใน `therapistLinks` สถานะ `approved` **ทันที** (ไม่มีสถานะ `pending` อีกต่อไป)
- ปุ่ม "ดำเนินการต่อ" → ไปขั้นตอนเพิ่มเบอร์ฉุกเฉิน (3.3b)
- ลิงก์ท้ายหน้า (edge case): "ไม่พบโรงพยาบาลหรือนักกายภาพบำบัดของคุณในรายการ? ติดต่อทีมงาน" — สำหรับกรณีข้อมูลยังไม่ครบในระบบ

### 3.3b `/onboarding/emergency-contact` (progressive, skippable)
แสดงถัดจากขั้นตอนเลือกนักกายภาพบำบัด (3.3)

**เนื้อหา:**
- ไอคอนใหญ่ + หัวข้อ "ยินดีต้อนรับสู่ RehabAI"
- คำอธิบาย: ชวนเพิ่มเบอร์ฉุกเฉิน บอกว่าใช้เวลาไม่ถึง 1 นาที และเพิ่มทีหลังได้ที่โปรไฟล์
- ฟอร์ม: ชื่อผู้ติดต่อฉุกเฉิน + ความสัมพันธ์ (select: บุตร/ธิดา, คู่สมรส, ญาติ, อื่นๆ) — 2 คอลัมน์, แล้วเบอร์โทรศัพท์ (เต็มความกว้าง)
- ปุ่มหลัก: "บันทึกและเข้าใช้งาน" → บันทึกลง `emergencyContacts` subcollection → redirect `/home`
- ลิงก์รอง (skip link): "ข้ามไปก่อน (เพิ่มทีหลังได้ที่หน้าโปรไฟล์)" → redirect `/home` โดยไม่บันทึก

### 3.4 `/home`
**Sidebar (ใช้ทุกหน้าในแอปผู้ป่วย):** โลโก้ RehabAI, เมนู 4 รายการ — หน้าแรก, ฝึกกายภาพบำบัด, สรุปผล, โปรไฟล์ (เมนูรอง เช่น การแจ้งเตือน/ผู้ติดต่อฉุกเฉิน/แชท ถูกย้ายไปรวมในหน้าโปรไฟล์แล้ว ไม่แสดงบน sidebar หลัก) + sidebar footer แสดง avatar/ชื่อผู้ใช้

**Topbar:** "สวัสดีตอนเช้า/บ่าย/เย็น" (ตามเวลาปัจจุบัน) + ชื่อผู้ใช้ | ไอคอนกระดิ่ง | avatar

**Content:**
- **Streak card** (พื้นหลัง teal-900 เข้ม): จำนวนวันฝึกต่อเนื่อง + คะแนนล่าสุด
- Section "เลือกโปรแกรมฝึกวันนี้"
- **Training card ×2** (grid 2 คอลัมน์):
  - LSVT BIG: ไอคอน, "ฝึกการเคลื่อนไหวร่างกาย ผ่านกล้องเว็บแคมพร้อมวิเคราะห์ท่าทางแบบเรียลไทม์", meta "20 นาที · 3 ท่าฝึก" (หรือจำนวนท่าที่เหลือวันนี้), ปุ่ม "เริ่มฝึก" → `/training/big`
  - LSVT LOUD: ไอคอน, "ฝึกความดังของเสียงพูด ผ่านไมโครโฟนพร้อมวัดระดับเสียงแบบเรียลไทม์", meta "10 นาที · 2 แบบฝึกหัด", ปุ่ม "เริ่มฝึก" → `/training/loud`

### 3.5 `/training/big` — รายการท่าฝึก LSVT BIG
**Topbar:** "ฝึกกายภาพบำบัด · เรียงจากศีรษะไปเท้า" | หัวข้อ "LSVT BIG — ฝึกการเคลื่อนไหว" | pill แสดงความคืบหน้า เช่น "2/8 ท่าเสร็จแล้ว"

**รายการท่าฝึก (เรียงลำดับตายตัวจากศีรษะลงเท้า — ห้ามสลับลำดับ):**

| # | ชื่อท่า | กลุ่ม | วิธีทำ | เป้าหมาย |
|---|---|---|---|---|
| 1 | หมุนไหล่ออก (Shoulder External Rotation) | stretch | ยืนตรง งอศอก 90° แนบลำตัว หมุนแขนออกด้านนอกช้าๆ ค้างไว้ 3-5 วิ แล้วหมุนกลับ | 10 ครั้ง |
| 2 | Overhead Reach (เอื้อมแขนเหนือศีรษะ) | stretch | ยืนตัวตรง ยกแขนสองข้างขึ้นเหนือศีรษะให้สุดช่วง ค้างไว้ 2 วิ แล้วลดลงช้าๆ | 10 ครั้ง |
| 3 | Lateral Raise (กางแขนด้านข้าง) | strength | กางแขนออกด้านข้างระดับไหล่ ค้างไว้ 3 วิ แล้วลดลง | 10 ครั้ง |
| 4 | ยกแขน-ขา (Arm-Leg Raise) | balance | คุกเข่า/ยืนทรงตัว ยกแขนขวา+ขาซ้ายพร้อมกันขนานพื้น ค้างไว้ 3 วิ สลับข้าง | 8-10 ครั้ง/ข้าง |
| 5 | ยกขา+มือแตะเข่าฝั่งตรงข้าม (Marching + Cross Touch) | balance | ยกเข่าขวาขึ้น มือซ้ายแตะเข่า สลับข้าง | 10 ครั้ง/ข้าง |
| 6 | Sit-to-Stand (นั่ง-ยืน) | strength | ลุกยืนจากเก้าอี้อย่างมั่นคง เน้นแรงขาและการทรงตัว | 10 ครั้ง |
| 7 | เขย่งเท้า (Heel Raise) | strength | ยืนตรง เขย่งปลายเท้าขึ้นสูงสุด ค้างไว้ 2 วิ แล้วลดส้นเท้าลง | 10 ครั้ง |
| 8 | ก้าวถ่ายน้ำหนัก (Weight Shift Step) | balance | ก้าวเท้าตามมาร์กบนพื้น ถ่ายน้ำหนักไปเท้าหน้า แล้วก้าวถอยกลับ ระบบวัดระยะปลายเท้าถึงปลายเท้าให้อยู่ 30-35 ซม. | 10 ครั้ง/ข้าง |

**UI element ต่อแถว (เป็นข้อความล้วน ไม่มีไอคอน/ภาพประกอบ):** ลำดับ (วงกลมเลข หรือ ✓ ถ้าเสร็จแล้ว), ชื่อท่า+badge กลุ่ม (สี stretch=teal, balance=coral, strength=เทา), คำอธิบายสั้น, meta จำนวนครั้งเป้าหมาย, badge สถานะ (เสร็จแล้ว/กำลังฝึก/ยังไม่เริ่ม), ปุ่ม "เริ่มฝึก" (ถ้ายังไม่เสร็จ) → ไป `/training/big/session?exercise={id}` (หรือ `/training/big/session-weightshift` สำหรับท่าที่ 8)

### 3.7 `/training/big/session?exercise=:id` — หน้าฝึกจริง (pose estimation)
**Layout:** 2 คอลัมน์ (ซ้าย 1.6fr กล้อง, ขวา 1fr ข้อมูล)

**คอลัมน์ซ้าย (cam-panel, พื้นหลัง `--cam-bg`):**
- Badge มุมซ้ายบน: "ครั้งที่ {n} / {target}"
- Badge มุมขวาบน: จุดแดงกระพริบ + "วิเคราะห์ท่าทางแบบเรียลไทม์"
- ปุ่ม SOS สีแดง กึ่งกลางด้านบน (ดู 3.13)
- ภาพจากกล้องจริง + overlay โครงร่าง pose skeleton (ใช้ pose estimation model วาดจุด/เส้นทับภาพ)
- แถบ feedback ด้านล่าง (สีเขียวถ้าท่าถูก): เช่น "ท่าถูกต้อง — ยกแขนได้สุดช่วงเคลื่อนไหวแล้ว"

**คอลัมน์ขวา (session-info-card):**
- ชื่อท่า + คำอธิบายวิธีทำ (ดึงจากตารางใน 3.5)
- Progress bar (ครั้งปัจจุบัน / เป้าหมาย)
- Stat rows: ความแม่นยำท่าทาง (%), เวลาในเซสชันนี้, คะแนนสะสมวันนี้
- ปุ่ม "หยุดฝึก" (outline สีแดง)

**Logic:** pose estimation ต้องตรวจจับมุมข้อต่อที่เกี่ยวข้องกับท่านั้นๆ (ไหล่/ศอกสำหรับ Overhead Reach, เข่า/สะโพกสำหรับ Sit-to-Stand ฯลฯ) นับ rep เมื่อครบ range of motion ที่กำหนด และคำนวณ % ความแม่นยำเทียบกับ range ที่ถูกต้อง

### 3.9 `/training/loud/session` — ฝึกจริง LSVT LOUD (voice meter)
**Layout:** 2 คอลัมน์ เหมือน 3.7 แต่คอลัมน์ซ้ายเป็น **mic-panel** แทนกล้อง:
- ปุ่ม SOS ด้านบน
- Phrase card: คำ/วลีที่ต้องออกเสียง เช่น "อาาาาา"
- ตัวเลขความดังขนาดใหญ่ (dB) — ใช้ Web Audio API วิเคราะห์ amplitude จากไมโครโฟน
- VU meter (แถบสี gradient) พร้อมเส้น target แสดงเป้าหมาย dB
- Caption: เบา / เป้าหมาย {n} dB / ดัง

**คอลัมน์ขวา:** เหมือน 3.7 แต่ stat เปลี่ยนเป็น: ความดังเฉลี่ย, ระยะเวลาเปล่งเสียงนานสุด, คะแนนสะสมวันนี้

**หมายเหตุทางเทคนิค:** ค่า dB ที่แสดงต้องผ่านการ**สอบเทียบ (calibration)** noise floor ของไมค์ก่อนใช้งานจริง — ฟีเจอร์นี้ยังไม่ได้ออกแบบละเอียด (อยู่ใน backlog หัวข้อ 7)

### 3.11 `/profile` — Hub รวมเมนูรอง
**Sections (เรียงบนลงล่าง):**

### 4.2 `/register` — เลือกโรงพยาบาล + ยืนยันตัวตน

1. **เลือกภาค** — dropdown 1 ค่า: ภาคกลาง, ภาคเหนือ, ภาคตะวันออกเฉียงเหนือ, ภาคใต้, ภาคตะวันออก, ภาคตะวันตก
2. **สังกัดโรงพยาบาล** — dropdown ที่ **options เปลี่ยนแบบเรียลไทม์ทันทีที่เปลี่ยนภาคในข้อ 1** (กลไกเดียวกับฝั่งผู้ป่วยในหน้า 3.3) ครอบคลุมโรงพยาบาลระดับจังหวัด/อำเภอทั่วประเทศ (ไม่ใช่แค่โรงพยาบาลใหญ่ในกรุงเทพฯ) รวมประมาณ 90+ แห่งในชุดข้อมูลตั้งต้น — แต่ละโรงพยาบาลที่มีข้อมูลผูก **โดเมนอีเมลองค์กร** ไว้ (เช่น ศิริราช = `si.mahidol.ac.th`) จะใช้ในการเช็คโดเมนอัตโนมัติ ส่วนโรงพยาบาลที่ยังไม่มีข้อมูลโดเมนจะข้ามไปที่ manual review ทั้งหมด
   - **ฟังก์ชันเพิ่มโรงพยาบาลใหม่ (hybrid: พิมพ์เอง + ตรวจสอบก่อนเผยแพร่)** — ลิงก์ "ไม่พบโรงพยาบาลของคุณในรายการ? เพิ่มโรงพยาบาลใหม่" ใต้ dropdown เปิดฟอร์มย่อยให้พิมพ์: ชื่อโรงพยาบาลใหม่, ภาค (dropdown), จังหวัด — ข้อมูลนี้จะถูกส่งเข้าคิว **"รอตรวจสอบ"** ไม่ปรากฏใน dropdown ของผู้ป่วยทันที ทีมงานตรวจสอบก่อนว่าไม่ซ้ำกับรายชื่อที่มีอยู่แล้ว (ป้องกันปัญหาสะกดต่างกันแต่เป็นที่เดียวกัน เช่น "รพ.เชียงรายประชานุเคราะห์" vs "โรงพยาบาลเชียงรายประชานุเคราะห์") ก่อนเผยแพร่เข้า `hospitals` collection จริง — เมื่อ approve แล้วจะกลายเป็นตัวเลือกถาวรให้คนถัดไปเลือกได้เลยโดยไม่ต้องพิมพ์ซ้ำ
3. **ตำแหน่งในโรงพยาบาล** — **text input พิมพ์เอง** (ไม่ใช่ตัวเลือกสำเร็จรูป) พร้อม placeholder ตัวอย่าง "เช่น นักกายภาพบำบัด, นักกายภาพบำบัดชำนาญการ" เนื่องจากแต่ละโรงพยาบาลใช้ชื่อตำแหน่ง/ระดับไม่เหมือนกัน (เช่น ระบบราชการมีตำแหน่งทางวิชาการเฉพาะ เช่น "ชำนาญการ", "ชำนาญการพิเศษ" ที่ต่างจากโรงพยาบาลเอกชน)
4. ชื่อ-นามสกุล
5. อีเมลที่ใช้งาน — **มีการเช็คโดเมนแบบเรียลไทม์** (ดูด้านล่าง)
6. รหัสผ่าน

**Backend logic (ขั้นที่ 1-2):** เก็บ `region` แยกจาก `hospitalId` ในฟอร์ม — เมื่อผู้ใช้เปลี่ยนภาค ให้ client query `hospitals` ที่ `region` ตรงกันมาเติม dropdown โรงพยาบาลใหม่ทั้งหมด (เหมือน logic เดียวกับหน้า 3.3 ฝั่งผู้ป่วย ใช้ dataset เดียวกันได้เลยเพื่อความสอดคล้อง)

**Real-time email domain check (client-side, ไม่ผูกกับการอนุมัติจริง):**
- ทันทีที่เลือกโรงพยาบาลหรือพิมพ์อีเมล ระบบเทียบโดเมนอีเมลที่กรอกกับโดเมนที่ผูกไว้กับโรงพยาบาลนั้น
- **ตรงกัน** → แสดงข้อความเขียว "อีเมลตรงกับโดเมนของ {โรงพยาบาล} — ผ่านการเช็คเบื้องต้นอัตโนมัติ"
- **ไม่ตรงกัน** → แสดงข้อความเหลือง "อีเมลนี้ไม่ตรงกับโดเมนของ {โรงพยาบาล} (@{domain}) ระบบจะส่งให้ทีมงานตรวจสอบเพิ่มเติมแบบ manual review"
- **โรงพยาบาลไม่มีข้อมูลโดเมนในระบบ** (เช่นคลินิกเล็ก) → แสดงข้อความกลาง "โรงพยาบาลนี้ยังไม่มีข้อมูลโดเมนอีเมลในระบบ จะใช้การตรวจสอบแบบ manual review ทั้งหมด"
- **สำคัญ:** การเช็คนี้เป็นแค่ **สัญญาณเตือนเบื้องต้น (soft signal) ไม่ใช่การอนุมัติอัตโนมัติ** — ผ่านหรือไม่ผ่านด่านนี้ก็ยังต้องรอ manual review เสมอ เพราะโดเมนอีเมลยืนยันได้แค่ว่า "เกี่ยวข้องกับโรงพยาบาล" ไม่ได้ยืนยันว่า "เป็นนักกายภาพบำบัดตัวจริงตามที่กรอก"

หลัง submit → สถานะ **"รอการตรวจสอบ"** (แถบเหลือง): อธิบายว่าการเช็คโดเมนอีเมลเป็นสัญญาณเบื้องต้นเท่านั้น ทีมงานจะติดต่อแผนกกายภาพบำบัดของโรงพยาบาลที่เลือกไว้เพื่อยืนยันตัวตนอีกครั้งเสมอก่อนเปิดใช้งานบัญชี ระบบจะแจ้งผลภายใน 1-2 วันทำการ

**Backend logic:** บัญชีสร้างในสถานะ `pending_verification` พร้อม `hospitalId`, `position`, และ `emailDomainMatch` (bool, ผลเช็คจาก client ส่งมาบันทึกไว้ให้ทีม manual review ใช้ประกอบการตัดสินใจ ไม่ใช่ตัวตัดสินสุดท้าย) — ต้องมีกระบวนการยืนยันตัวตนกับโรงพยาบาลนั้นจริงจากทีมงาน (โทร/อีเมลไปยังแผนกกายภาพบำบัด) ก่อนเปลี่ยนสถานะเป็น `active` เสมอ ไม่ว่าผลเช็คโดเมนจะเป็นอย่างไร กรณีเพิ่มโรงพยาบาลใหม่ ต้อง insert doc ใน `hospitals` ด้วยสถานะ `pending` แยกต่างหาก และห้าม query มาแสดงใน dropdown ของหน้า 3.3 (ฝั่งผู้ป่วย) จนกว่าจะเปลี่ยนเป็น `approved`

### 4.3 `/patients` — รายชื่อผู้ป่วย
- แถบข้อความอธิบาย (สีเขียว): ผู้ป่วยที่เลือกนักกายภาพบำบัดคนนี้ตอน onboarding (3.3) จะปรากฏในรายชื่อนี้อัตโนมัติทันที **ไม่มีปุ่ม/ฟอร์มให้นักกายภาพบำบัด "เพิ่มผู้ป่วย" เองอีกต่อไป** — การเชื่อมต่อเกิดจากฝั่งผู้ป่วยเลือกเท่านั้น
- Search bar ค้นหาชื่อผู้ป่วย
- Patient row ต่อคน: avatar, ชื่อ, ระยะอาการ, เวลาฝึกล่าสุด, meta (ต่อเนื่องกี่วัน/คะแนนล่าสุด/สัปดาห์นี้กี่ครั้ง), ปุ่ม "ดูรายละเอียด" + "แชท"

### 4.4 `/patients/:id` — รายละเอียดผู้ป่วย (read-only, ห้ามมีปุ่มแก้ไขโปรแกรมฝึก)
- Breadcrumb กลับไปรายชื่อผู้ป่วย + ปุ่ม "แชทกับผู้ป่วย"
- Metric row: คะแนนสะสม, จำนวนครั้งที่ฝึก, ฝึกต่อเนื่อง, วันที่ฝึกสัปดาห์นี้
- Chart: ความถี่ฝึก 7 วัน (BIG vs LOUD) — เหมือน dashboard ผู้ป่วย
- **ความแม่นยำแยกตามท่าฝึก:** progress bar ต่อท่า (ทั้ง 8 ท่าของ LSVT BIG) แสดง % ความแม่นยำเฉลี่ย — ช่วยให้เห็นว่าท่าไหนผู้ป่วยทำได้แย่เป็นพิเศษ
- ตารางประวัติการฝึกล่าสุด
- **บันทึกของฉัน:** textarea ส่วนตัว เห็นเฉพาะนักกายภาพบำบัดคนนั้น ไม่กระทบหรือแสดงในฝั่งผู้ป่วย — ปุ่มบันทึก

**ข้อจำกัดสำคัญที่ต้องคุมใน dev:** หน้านี้ต้องไม่มีการ mutate ข้อมูลโปรแกรมฝึกของผู้ป่วยเด็ดขาด (ไม่มีปุ่มเพิ่ม/ลด/แก้ไขท่าฝึก) — ทำได้แค่ดูข้อมูลและเขียนบันทึกส่วนตัว

### 4.5 `/messages` (ฝั่งนักกายภาพบำบัด)
- รายชื่อการสนทนาทุกผู้ป่วย (ซ้าย) + หน้าต่างแชท (ขวา)
- แถบคำเตือนใต้ header เดียวกับฝั่งผู้ป่วย (3.16): "ช่องทางนี้ไม่ใช่สำหรับเหตุฉุกเฉิน ระบบจะแจ้งเตือนให้ตอบภายใน 1-2 วันทำการ"
- ปุ่ม "ดูความคืบหน้า" ในแชท → ลิงก์ไป `/patients/:id` ของคู่สนทนานั้น

```
users/{uid}
  - name, email, phone, parkinsonStage ("stage1" | "stage2")
  - role: "patient"
  - createdAt

  - enabled (bool), days ([mon,tue,...]), time ("09:00"), condition ("always"|"missedOnly"|"streakBroken")

users/{uid}/therapistLinks/{therapistId}
  - status: "approved" | "revoked"   // ไม่มีสถานะ "pending" อีกต่อไป — เชื่อมต่อทันทีที่ผู้ป่วยเลือกจากรายการ (3.3)
  - linkedAt, revokedAt

users/{uid}/chats/{therapistId}/messages/{messageId}
  - senderId, senderRole, text, timestamp

therapists/{tid}
  - position (free-text string, พิมพ์เองตอนสมัคร — ไม่ใช่ enum เพราะแต่ละโรงพยาบาลใช้ชื่อตำแหน่งต่างกัน), name, email, hospitalId, hospitalName
  - emailDomainMatch (bool)   // ผลเช็คโดเมนอัตโนมัติตอนสมัคร ใช้ประกอบ manual review เท่านั้น ไม่ใช่ตัวตัดสิน
  - verificationStatus: "pending" | "active" | "rejected"   // เฉพาะ "active" เท่านั้นที่จะปรากฏให้ผู้ป่วยเลือกได้ในหน้า 3.3
  - role: "therapist"

therapists/{tid}/notes/{patientId}
  - text, updatedAt   // บันทึกส่วนตัว เห็นเฉพาะนักกายภาพบำบัดคนนี้

hospitals/{hospitalId}

  - status: "approved" | "pending"   // "pending" = เพิ่งถูกเพิ่มโดยนักกายภาพบำบัดผ่านฟอร์ม "เพิ่มโรงพยาบาลใหม่" (4.2) ยังไม่ผ่านการตรวจสอบ
  // region ใช้ filter หน้า 3.3 ขั้นที่ 2 (เลือกโรงพยาบาลเฉพาะในภาคที่เลือก)
  // emailDomain ใช้เทียบกับอีเมลที่กรอกตอนนักกายภาพบำบัดสมัคร (4.2); ถ้าไม่มีค่านี้ ระบบข้ามการเช็คอัตโนมัติไปที่ manual review เลย
  // หน้า 3.3 (ฝั่งผู้ป่วย) query เฉพาะ status == "approved" เท่านั้น — โรงพยาบาลที่เพิ่งเพิ่มใหม่ (pending) จะไม่ปรากฏใน dropdown จนกว่าทีมงานจะ approve
```

## 6. Key Flows Summary

### 6.1 การเชื่อมต่อผู้ป่วย-นักกายภาพบำบัด (tap-select, auto-link — ไม่มีรหัสหรือการรออนุมัติ)
**สมมติฐานหลัก:** ผู้ป่วยมาใช้แอปเพราะนักกายภาพบำบัดแนะนำ = เคยพบตัวจริงมาแล้วอย่างน้อย 1 ครั้ง ความสัมพันธ์นี้เองคือการยืนยันตัวตน จึงไม่ต้องมีขั้นตอนรอผู้ป่วยอนุมัติคำขอแบบระบบเดิม

1. หลังสมัครสมาชิก ผู้ป่วยเข้าสู่หน้า `/onboarding/select-therapist` (3.3)
2. เลือกภาค → ระบบ filter โรงพยาบาลใน `hospitals` ที่ `region` ตรงกัน
3. เลือกโรงพยาบาล → ระบบ filter นักกายภาพบำบัดใน `therapists` ที่ `hospitalId` ตรงกัน **และ** `verificationStatus == "active"` เท่านั้น
4. เลือกนักกายภาพบำบัด 1 คน (กดที่การ์ด) → ระบบสร้าง doc ใน `users/{uid}/therapistLinks/{therapistId}` สถานะ `approved` **ทันที**
5. นักกายภาพบำบัดเห็นผู้ป่วยคนนี้ปรากฏในหน้า `/patients` ของตนเองโดยอัตโนมัติ ไม่ต้องทำอะไรเพิ่ม
6. ผู้ป่วยกด "เปลี่ยนนักกายภาพบำบัด" หรือ "ยกเลิกการเชื่อมต่อ" ในหน้าโปรไฟล์ได้ทุกเมื่อ → สถานะเปลี่ยนเป็น `revoked` → นักกายภาพบำบัดเข้าถึงข้อมูลไม่ได้ทันที

**ข้อควรระวังตอน implement:** ต้องมี query `where('hospitalId','==',selectedHospitalId).where('verificationStatus','==','active')` ตอนดึงรายชื่อนักกายภาพบำบัดในขั้นที่ 3 เสมอ เพื่อไม่ให้นักกายภาพบำบัดที่ยังไม่ผ่านการตรวจสอบ (`pending`/`rejected`) ปรากฏให้ผู้ป่วยเลือกได้

### 6.2 Emergency detection flow
ดูหัวข้อ 3.15 — ลำดับต้องตรงตามนั้นเป๊ะ (แจ้งผู้ติดต่อทันที → นับถอยหลังก่อนโทร 1669 → ยกเลิกได้)

### 6.3 SMS reminder
Cloud Function (cron ทุกชั่วโมง) → query users ที่ `smsPreferences.enabled = true` และวัน/เวลาตรงกับตอนนี้ → เช็คเงื่อนไข (ฝึกวันนี้ยังไหม/ขาดฝึกกี่วัน) → เรียก SMS gateway ส่งข้อความ

## 7. Backlog (ยังไม่ได้ออกแบบ UI ในเฟสนี้)

- **Pre/Post-test คลินิก** (UPDRS, gait speed, speech intelligibility) — สำหรับวัดผลลัพธ์ทางคลินิกจริง แทนคะแนนแบบเกม
- **Calibration** ก่อนฝึกจริง (ตั้งกล้อง, วัด noise floor ของไมค์เพื่อให้ค่า dB แม่นยำ)
- **Contraindication warning** ก่อนเริ่มแต่ละท่า ตามปัญหาสุขภาพร่วมเฉพาะบุคคล

**สิ่งที่ตัดออกจากสโคปแล้ว (อย่าทำ):** Fall-risk screening (TUG/BBS), Caregiver mode เต็มรูปแบบ — เนื่องจากกลุ่มเป้าหมายเป็นผู้ป่วยระยะ 1-2 ที่ช่วยเหลือตนเองได้
